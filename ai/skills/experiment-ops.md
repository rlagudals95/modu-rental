# Skill: Experiment Ops

## Use when

- 새 실험을 등록할 때
- 랜딩 페이지 카피를 바꿀 때
- 리드 폼과 상담 폼 사이의 경계를 조정할 때
- success metric을 정할 때
- 이벤트 이름이나 CTA 이름을 정할 때
- 어떤 신호를 이벤트로 남겨야 할지 결정할 때
- 실험 종료 여부를 판단할 때

## Workflow

1. 검증하려는 고객 문제를 한 문장으로 쓴다.
2. 바꾸려는 자극을 한 가지로 제한한다.
3. 기대 행동을 이벤트 또는 전환으로 쓴다.
4. success metric을 비교 가능한 식으로 작성한다.
5. 리드와 상담 요청처럼 다른 강도의 신호를 같은 bucket으로 섞지 않는다.
6. 실험 종료 후 다음 액션이 무엇인지 미리 적는다.

## PMF Signal Checklist

1. 이 변경이 더 강한 PMF 신호를 얻는가, 아니면 단순 구현만 늘리는가?
2. 리드와 상담 요청을 같은 bucket으로 섞고 있지 않은가?
3. 실험의 종료 조건과 success metric이 문서화되어 있는가?
4. 다음 제품에서도 재사용 가능한 추상화인가?
5. 지금 당장 인터뷰/후속 연락 우선순위 결정에 도움이 되는 데이터인가?

## Guardrails

- 카피, 채널, 오퍼, 폼 구조를 한 번에 다 바꾸지 않는다.
- 이벤트는 필요한 것만 남긴다.
- `new`, `tmp`, `final` 같은 이름은 쓰지 않는다.
- 실험 설명보다 후속 행동 기준이 더 중요하다.
- PMF 신호를 흐리는 계측이나 funnel 단순화는 피한다.
