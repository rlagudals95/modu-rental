# Architecture Decisions & Tradeoffs

## 목표

이 저장소는 단일 제품 코드베이스가 아니라 **PMF 실험 재사용 보일러플레이트**다.
즉, 빠른 실험 속도와 구조적 재사용성 사이의 균형을 의도했다.

## 주요 선택

### 1) Monorepo (pnpm + Turbo)
- **선택 이유:** 제품 앱 + 공유 패키지(UI/도메인/DB/분석) 공진화가 쉬움
- **트레이드오프:** 초기 설정 복잡도 증가
- **완화:** 패키지 수를 7개로 제한하고 책임을 명확히 분리

### 2) Next.js 단일 앱 아키텍처
- **선택 이유:** API 서버 분리 없이 App Router만으로 실험 속도 확보
- **트레이드오프:** 경계가 흐려질 수 있음
- **완화:** `packages/core`, `packages/db`, `packages/analytics`로 관심사 분리

### 3) Supabase + Drizzle
- **선택 이유:** Postgres 기반 + Auth 확장성 + 타입 안전 스키마
- **트레이드오프:** 초기에는 인메모리 대비 무겁다
- **완화:** 현재는 mock store로 시작하고, 필요한 시점에 DB 스위치

### 4) Analytics abstraction
- **선택 이유:** 벤더 락인 방지. `track()` 하나로 시작
- **트레이드오프:** 고급 분석 기능 부재
- **완화:** Provider 인터페이스로 점진 확장

### 5) 테스트 전략: unit + smoke E2E
- **선택 이유:** trunk-friendly 최소 안정성 확보
- **트레이드오프:** 깊은 회귀 방어는 약함
- **완화:** 핵심 폼 검증 로직 + 대표 유저 플로우만 우선 커버

## 도메인 모델 설계 원칙

초기 모델은 특정 렌탈 업종에 과적합되지 않도록 일반화했다.
- leads
- consultation_requests
- products
- experiments
- page_events

다음 실험 제품으로 이동할 때도 이 5개 축은 대부분 재사용 가능하다.

## 비의도적 복잡성 방지

의도적으로 제외한 것:
- 분리 API 서버 / 마이크로서비스
- 큐/비동기 잡
- 복잡한 권한 시스템
- 외부 분석 벤더 의존

PMF 단계에서 중요한 것은 기능 완성도보다 학습 속도다.
